# <a name="data-matching-for-subject-rights-requests"></a>Сопоставление данных для запросов прав субъекта

С помощью сопоставления данных организации могут включить Microsoft Priva для идентификации субъектов данных на основе точных значений данных. Это может помочь повысить точность поиска контента субъекта данных, соответствующего этим значениям данных как для внутреннего персонала, так и для внешних пользователей, с которыми вы взаимодействуете. Он также упрощает предоставление полей вручную во время создания запроса прав субъекта, а также предоставляет контекст в запросах прав субъекта и для плитки "Обзор", демонстрирующий элементы с наиболее важным содержимым субъекта данных. Дополнительные сведения об этом представлении см. в статье ["Поиск и визуализация персональных данных в Priva"](priva-data-profile.md#items-with-the-most-data-subject-content).

Чтобы использовать функцию сопоставления данных, необходимо быть членом группы ролей "Управление конфиденциальностью". В Priva на портале [соответствия Требованиям Microsoft Purview](https://compliance.microsoft.com/) выберите Параметры  в верхней области навигации, а затем сопоставление **данных**. Здесь необходимо определить схему персональных данных и предоставить отправку персональных данных, как показано ниже. Обратите внимание, что вы можете добавлять элементы и удалять добавляемые элементы, но не можете изменять их.

## <a name="prepare-for-data-import"></a>Подготовка к импорту данных

Перед определением схемы или отправкой данных необходимо определить источник сведений о субъекте данных. Требуемый формат файла .csv, который может считываться приложением, например Microsoft Excel. Структурируйте этот экспорт таким образом, чтобы заголовки столбцов отображались в первой строке. Эти заголовки должны содержать имена атрибутов для схемы персональных данных. Проверьте формат данных в каждом поле. Если какой-либо из данных содержит запятые, заключите эти значения в двойные кавычки, чтобы убедиться, что они не будут анализироваться в отдельные поля.

## <a name="define-the-personal-data-schema"></a>Определение схемы персональных данных

Первым шагом в настройке сопоставления данных является определение схемы персональных данных, которая будет описывать атрибуты для субъектов данных. Вы загрузите эту схему на первой вкладке в области параметров сопоставления данных. Необходимые файлы **включают** XML-файл схемы персональных данных и XML-файл **пакета** правил.

### <a name="personal-data-schema-xml"></a>XML-код схемы персональных данных

Файл схемы персональных данных — это XML-файл, который определяет ожидаемые имена столбцов.

- Приведите имя файла *схемыpdm.xml*.
- Определите имя каждого столбца с помощью тега "Имя поля", как показано в примере ниже.
- Используйте значение "true" для полей, для которых требуется выполнить поиск, не более пяти полей. По крайней мере одно из имен полей должно быть доступны для поиска. Пример синтаксиса: `\<Field name="" searchable=""/>`.
- Схема персональных данных содержит раздел тега DataStore. Четыре обязательных поля должны быть сопоставлены с именами полей: primaryKeyField, upnField, firstNameField, lastNameField.

Например, следующий XML-файл определяет пример схемы с пятью полями, которые можно найти: PatientID, MRN, SSN, Телефон и DOB. PrimaryKeyField сопоставляется с PatientID, upnField сопоставляется с MRN, firstNameField сопоставляется с FirstName, а lastNameField сопоставляется с LastName.

Вы можете скопировать, изменить и использовать наш пример.

 ```xml
<PdmSchema xmlns="http://schemas.microsoft.com/office/2020/pdm">
      <DataStore name="Patientrecords" description="Schema for patient records" version="1" primaryKeyField="PatientID" upnField="MRN" firstNameField="FirstName" lastNameField="LastName">
            <Field name="PatientID" searchable="true"/>
            <Field name="MRN" searchable="true" />
            <Field name="FirstName" />
            <Field name="LastName" />
            <Field name="SSN" searchable="true" />
            <Field name="Phone" searchable="true" />
            <Field name="DOB" searchable="true" />
            <Field name="Gender" />
            <Field name="Address" />
      </DataStore>
</PdmSchema>
 ```

### <a name="rule-package-xml"></a>XML-код пакета правил

При настройке пакета правил убедитесь, что вы правильно ссылались на созданный выше файл схемы персональных данных: pdm.xml. В следующем примере XML-кода пакета правил необходимо настроить следующие поля, чтобы создать конфиденциальный тип данных, соответствующий данным:

- **Идентификатор RulePack** &  **Идентификатор PrivacyMatch**: используйте new-GUID для создания GUID.
- **Хранилище данных**: в этом поле указывается хранилище данных подстановки для сопоставления персональных данных, которое будет использоваться. Укажите определенное имя DataStore для настроенной схемы персональных данных.
- **idMatch**: это поле указывает на первичный элемент для сопоставления персональных данных.
  - **Соответствует**: указывает поле, которое будет использоваться при точном поиске. Укажите имя поля, доступного для поиска, из схемы персональных данных.
  - **Классификация**. В этом поле указывается соответствие конфиденциального типа, которое активирует поиск соответствия персональных данных. Вы можете указать имя или идентификатор GUID имеющегося встроенного или настраиваемого типа конфиденциальной информации. Чтобы избежать проблем с производительностью, при использовании пользовательского типа конфиденциальной информации в качестве элемента classification в сопоставлении персональных данных не используйте пользовательский тип конфиденциальной информации, который будет соответствовать большому проценту содержимого (например, "любое число" или "любое пятибуквенное слово"). Рекомендуется добавить вспомогательные ключевые слова или включить форматирование в определение настраиваемого типа конфиденциальной информации классификации.
- **Совпадение**: это поле указывает на дополнительные свидетельства, найденные в непосредственной близости от idMatch.
  - **Соответствует:** укажите любое имя поля в схеме персональных данных для DataStore.
- **Ресурс**: в этом разделе указывается имя и описание конфиденциального типа в нескольких языковых стандартах.
  - **idRef**: укажите GUID для идентификатора ExactMatch.
  - **Имя & описаний**: настройте по мере необходимости.

В приведенном ниже примере XML пакета правил мы ссылаемся на файл pdm.xml из предыдущего шага, который создает XML-файл схемы персональных данных:

- **Хранилище данных**: имя dataStore ссылается на ранее созданный файл схемы: dataStore = "PatientRecords".
- **idMatch**: значение idMatch ссылается на поле с возможностью поиска, которое указано в созданном ранее pdm.xml файле: idMatch соответствует ="SSN".
  - **Классификация**: значение классификации ссылается на существующий или пользовательский тип конфиденциальной информации: classification = "Номер социального страхования США (SSN)". (В этом случае используется существующий тип конфиденциальной информации, содержащий номер социального страхования США.)

Создайте пакет правил в формате XML (с кодировка Юникода), как показано в следующем примере кода. Этот пример можно скопировать, изменить и использовать.

 ```xml
<RulePackage xmlns="http://schemas.microsoft.com/office/2020/pdm">
  <RulePack id="fd098e03-1796-41a5-8ab6-198c93c62b21">
    <Version build="0" major="2" minor="0" revision="0" />
    <Publisher id="eb553734-8306-44b4-9ad5-c388ad970528" />
    <Details defaultLangCode="en-us">
      <LocalizedDetails langcode="en-us">
        <PublisherName>IP DLP</PublisherName>
        <Name>Health Care PDM Rulepack</Name>
        <Description>This rule package contains the Personal Data Match sensitive type for health care sensitive types.</Description>
      </LocalizedDetails>
    </Details>
  </RulePack>
  <Rules>
    <PrivacyMatch id = "E1CC861E-3FE9-4A58-82DF-4BD259EAB381" patternsProximity = "300" dataStore ="PatientRecords" recommendedConfidence = "65" >
      <Pattern confidenceLevel="65">
        <idMatch matches = "SSN" classification = "U.S. Social Security Number (SSN)" />
      </Pattern>
      <Pattern confidenceLevel="75">
        <idMatch matches = "SSN" classification = "U.S. Social Security Number (SSN)" />
        <Any minMatches ="3" maxMatches ="6">
          <match matches="PatientID" />
          <match matches="MRN"/>
          <match matches="FirstName"/>
          <match matches="LastName"/>
          <match matches="Phone"/>
          <match matches="DOB"/>
        </Any>
      </Pattern>
    </PrivacyMatch>
    <LocalizedStrings>
      <Resource idRef="E1CC861E-3FE9-4A58-82DF-4BD259EAB381">
        <Name default="true" langcode="en-us">Patient SSN Exact Match.</Name>
        <Description default="true" langcode="en-us">PDM Sensitive type for detecting Patient SSN.</Description>
      </Resource>
    </LocalizedStrings>
  </Rules>
</RulePackage>
 ```

## <a name="sensitive-info-types"></a>Типы конфиденциальной информации

Второй этап настройки сопоставления данных — создание уникальных типов конфиденциальной информации для сопоставления персональных данных (PDM). [Типы конфиденциальной информации (SIT)](/microsoft-365/compliance/sensitive-information-type-learn-about) — это классификаторы на основе шаблонов, которые обнаруживают конфиденциальную информацию, например номера социального страхования или кредитных карт. Настройка типа конфиденциальной информации PDM позволяет использовать точные значения данных, а не универсальные значения для обнаружения совпадений. Чтобы начать этот шаг, выберите "Создать тип конфиденциальной информации **PDM** ", чтобы запустить мастер создания.

## <a name="upload-personal-data"></a>Upload персональные данные

После определения схемы персональных данных и типов конфиденциальной информации третьим шагом является отправка персональных данных. Перейдите **на** вкладку "Отправка персональных данных", выберите "Добавить" и выберите личную схему, определенную на первом шаге, а затем отправьте файл, содержащий персональные данные.

Вы можете отправить эти персональные данные, выбрав локальный файл или укажите URL-адрес SAS для существующего служба хранилища Microsoft Azure, содержащего ваш файл персональных данных.
Если вы подготовили файл в качестве первого шага в этом процессе, соответствующего созданной схеме, вы можете использовать этот файл для отправки.
