# <a name="generate-reports-and-fulfill-a-subject-rights-request"></a>Создание отчетов и выполнение запроса на права субъекта

После завершения проверки данных для запроса на права субъекта в Microsoft Priva можно перейти к выполнению запроса. Priva создает отчеты и собирает файлы, помеченные как **Include** во время процесса проверки данных. Выбранные файлы из этих пакетов данных могут быть отправлены в субъект данных для выполнения их запроса. Priva также поддерживает использование API Microsoft 365 прав субъекта для внедрения возможностей автоматизации.

## <a name="understanding-reports"></a>Понимание отчетов

После выбора **полного просмотра** на этапе **проверки** данных запроса на права субъекта окончательные отчеты для запроса начнут создаваться автоматически. На **вкладке Отчеты** на странице Запросы на права субъекта в столбце Состояние указывается, когда происходит генерация отчетов и когда отчет готов **к скачивке**.  Создание отчетов может занять до 30 минут.

Отчеты делятся на два раздела:
1. **Отчеты для субъекта** данных. Эти отчеты содержат сведения, которые могут быть возвращены субъекту данных в рамках выполнения запроса. Здесь вы найдете пакет данных, содержащий файлы  для отправки в субъект данных.
   > [!IMPORTANT]
   > Пакет данных будет создан только в том случае, если вы пометите элементы **как Include** во время проверки данных.

   > [!IMPORTANT]
   > Пакет данных будет создан только для типов запросов **на** экспорт  и доступ. Пакет данных не будет создан для списка **tagged для последующего запроса** . Просмотрите сведения [о типах запросов на права субъекта](subject-rights-requests-create.md#use-the-subject-rights-request-creation-wizard).

2. **Отчеты для внутреннего использования**. Эти отчеты являются внутренними записями организации, связанными с запросом на права субъекта. Они включают журнал аудита и список всех файлов, к которые вы применяли теги во время проверки данных для последующей проверки или принятия дальнейших действий.

> [!NOTE]
> Когда создаются отчеты, организация завершает запрос, отправляя соответствующие отчеты безопасным способом субъекту данных для выполнения запроса на права субъекта. Если субъект данных запросил удаление их данных, организация несет ответственность за определение элементов для удаления и их удаление.

## <a name="data-package"></a>Пакет данных

Пакет данных запроса прав субъекта содержит элементы, помеченные как **Include** на этапе проверки данных процесса. Пакет данных создается как почтовый файл. Когда он будет готов к скачивке, выберите строку отчета на вкладке **Отчеты** на странице сведения о запросе прав субъекта. Панель вылетов откроется кнопкой для **скачивания** отчета. Следующие **инструкции на** панели вылетов предоставляют краткие ссылки на работу с содержимым.

Когда вы будете готовы скачать пакет данных, выберите **Скачивание**, найдите скачивание и откройте скачаный почтовый файл.

### <a name="contents-of-the-data-package"></a>Содержимое пакета данных

Почтовый файл пакета данных содержит следующие элементы:

- Папка с **именем Azure**. Эта папка содержит ключевые материалы для субъекта данных. Подробные пояснения см. ниже.
- Файлы за пределами **папки Azure** : они могут включать две электронные таблицы с именем **Results** и **Summary**. Эти файлы вне папки Azure предоставляются для справки и должны использоваться главным образом администраторами организации.

Когда Прива определяет и извлекает файлы для запроса на права субъекта, экспортируемые в этом процессе файлы переименовываются с помощью уникальных идентификаторов для защиты персональных данных. Вы можете перекрестно ссылаться на уникальные имена с исходными именами файлов с помощью **Export_load_file.csv** файла. Так как исходные имена файлов могут включать конфиденциальные сведения, следует следовать политикам, которые применяются к таким сведениям.

> [!NOTE]
> Рекомендуется тщательно просмотреть содержимое этой папки и отправку в субъект данных только необходимых файлов. Вы должны оценить свой ответ, чтобы убедиться, что он с учетом требований применимого законодательства.

### <a name="azure-folder-contents"></a>Содержимое папки Azure

Откройте **папку Azure** , затем **откройтеAEDExport.zip** файл, содержащий другую папку с длинным именем, состоящую из цифр и букв. Откройте папку с длинным именем, чтобы открыть содержимое, описанное ниже:

- **Extracted_text_files** папка: содержит текст, извлеченный из родных файлов (где поддерживается).
- **Папка NativeFiles** . Содержит все элементы, отмеченные во время проверки данных как **Включенные**, в их родном формате файлов. Каждому элементу в этой папке предоставляется уникальное имя файла для защиты персональных данных. Вы можете перекрестно ссылаться на эти уникальные имена с исходным именем файла Export_load_file.csv файл, указанный ниже.
  - Файлы, которые были отредактировали с помощью функции Аннотации во время процесса проверки данных, находятся в папке **NativeFiles** и имеют суффикс "_burn.pdf".
- **Export_load_file.csv** : содержит исходные имена файлов, чтобы можно было перекрестно ссылаться на них с уникальными именами файлов, предоставленными в папке NativeFiles.
- **Сводный** текстовый файл. Предоставляет сводку типов файлов в пакете данных, количество файлов и их размер.

##### <a name="what-to-do-with-the-folder-contents"></a>Что делать с содержимым папки

Откройте **папку Azure** и почтовые файлы, как было объяснено выше. Следующая задача — просмотреть элементы, определить, что отправлять субъекту данных, и при необходимости изменить содержимое почтового файла.

Например, для запроса на экспорт, в котором субъект данных хочет получить копию всех элементов, содержащих персональные данные организации, необходимо внимательно просмотреть элементы в папке **NativeFiles** и удалить любой элемент, который не хотите отправлять обратно в субъект данных.

Для запроса доступа, в котором субъект данных хочет получить сводку всех своих персональных данных, хранимой в организации, необходимо сосредоточиться на текстовом файле **Сводки** .

Измените почтовый файл, чтобы удалить содержимое, которое не нужно включать в конечный пакет. После завершения отправьте почтовый файл безопасным способом субъекту данных, который сделал запрос на права субъекта.

## <a name="audit-log"></a>Журнал аудита

Журнал аудита — это CSV-файл, предоставляющий отчет о прогрессировании этапов для каждого запроса на права субъекта. В нем перечислены каждый этап процесса, а также дата и время завершения, а также имя пользователя, завершившего шаг.

## <a name="tagged-files-reports"></a>Помеченные отчеты о файлах

Отчеты, помеченные как **Файлы** с тегами, являются CSV-файлами, в которых перечисляются элементы контента, к которым теги были применены во время процесса проверки данных. Теги используются для флага элементов контента для дополнительного внимания или действий организации. Дополнительные новости [о параметрах тегов](priva-settings.md#data-review-tags).

## <a name="retention-periods-for-reports-and-data"></a>Периоды хранения отчетов и данных

Отчеты, созданные запросами на права субъекта и связанными данными, такими как аннотированные файлы, сохраненные в Azure, хранятся в течение определенного времени. Срок хранения по умолчанию составляет 30 дней с даты закрытия запроса на права субъекта.

Период хранения данных определяется в Параметры и применяется ко  всем запросам на права субъекта. Чтобы просмотреть или изменить период хранения данных, выполните следующие действия:

1. В любом месте в запросах на права субъекта **Привы выберите** Параметры (значок шестеренки) в правом верхнем углу экрана.
2. Выберите **периоды хранения данных в** левой навигации.
3. В меню **"Данные, собранные и отчеты** " выберите 30 дней или 90 дней в качестве периода хранения.
4. Выберите **Сохранить** , чтобы сохранить параметры.

Убедитесь, что выбранные периоды хранения данных соответствуют политикам и юридическим обязательствам организации.

## <a name="integrate-with-partner-solutions"></a>Интеграция с решениями партнеров

Решение "Запросы на права субъекта Привы" можно интегрировать с существующими бизнес-процессами и инструментами, используя API запроса Microsoft 365 субъекта. Это предлагает простой, но мощный способ внедрения автоматизации в стратегию прав субъекта.

Когда субъекты данных запрашивают сведения в организации, вы можете использовать наши API для создания этих запросов в Microsoft 365 на основе настраиваемых для этого запроса критериев. Вы можете создать запрос на права субъекта в Microsoft 365, отслеживать ход выполнения запроса через его этапы, а также подтвердить, когда запрос завершена обработка и содержимое готово к извлечению. Наши API доступны для всех, кто использует, чтобы сделать их решения более доступными: будь то для isVs, партнеров для размещения Microsoft 365 в своих решениях, или для организаций, чтобы использовать с их линией бизнес-приложений.

Дополнительные дополнительные дополнительные информации см. в [Graph API запроса на права субъекта Майкрософт](/graph/api/resources/subjectrightsrequest-subjectrightsrequestapioverview).
